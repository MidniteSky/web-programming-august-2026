The framework landscape

A map of the big names - read-only. Nothing here runs.

These snippets do not work in this file. They're printed as text so you can recognise the style. Frameworks need a build step (Node, npm, a bundler) that's well beyond this course.

Imperative (what you know) vs declarative (frameworks)

The vanilla way - step-by-step DOM instructions, the Day 3-4 style:

// find the element, then change it, by hand
const count = document.querySelector("#count");
let n = 0;
button.addEventListener("click", () => {
    n = n + 1;
    count.textContent = n;   // you update the DOM yourself
});

The declarative way - describe the result; the framework updates the DOM:

// React: describe the UI for the current state; React does the DOM edits
function Counter() {
    const [n, setN] = useState(0);
    return <button onClick={() => setN(n + 1)}>{n}</button>;
}

The big three, side by side

React

The most widely used, backed by Meta. Huge job market and ecosystem. Uses "JSX" - HTML-like syntax written inside JavaScript.

function Hello({ name }) {
    return <p>Hello, {name}!</p>;
}

Vue

Approachable, often praised for gentle onboarding. Keeps HTML, CSS and JS in one "single-file component" that looks close to what you know.

<template>
    <p>Hello, {{ name }}!</p>
</template>

<script setup>
    const name = "Alice";
</script>

Svelte

Newer; compiles your components away to small, plain JavaScript at build time, so there's little framework left at runtime. Very tidy syntax.

<script>
    let name = "Bob";
</script>

<p>Hello, {name}!</p>

Angular also exists - older, larger, common in big corporate apps; a full framework rather than a library.

When do you actually need one?

Reach for a framework when...

  • the app is large and full of interactive state that changes a lot (dashboards, editors, live data)
  • a team needs a shared structure and reusable components across many screens
  • you're maintaining something already built in one

You probably don't need one for...

  • a handful of pages, a blog, a brochure site, a form
  • anything the vanilla DOM handles comfortably - like the Member Manager you just built
  • learning: fundamentals first, frameworks later
The honest take: frameworks solve a real problem - keeping a big, stateful UI in sync - but they add a build step, dependencies and concepts on top of the fundamentals. Because you now understand what the DOM actually does, learning one later will be far easier: you'll know what it's doing for you.
